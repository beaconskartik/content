1 Overview

Design a subscription service for Airbnb that allows guests and hosts to subscribe to monthly or annual plans with tiered pricing. Subscribers gain access to premium listings, booking discounts, priority customer support, and exclusive concierge experiences. The system must handle billing lifecycle management, entitlement gating, multi-currency support, and integrate with third-party payment processors like Stripe.

Why This Problem Matters at PE Level A subscription system cuts across payments, identity, entitlements, and user experience. At principal engineer scope, you must reason about distributed state machines, payment webhook reliability, eventual consistency in entitlement propagation, and cross-team API contracts. This is not a CRUD problem -- it is a financial-grade distributed system.

2 Requirements

Functional Requirements

  • Subscribe/Unsubscribe: Users can subscribe to any tier, upgrade, downgrade, or cancel at any time with prorated billing.
  • Tier Management: Admin CRUD for plans with versioning. Existing subscribers grandfathered on price changes.
  • Premium Access Control: Gate premium listings, discounts, and features by active entitlement.
  • Billing Cycles: Monthly and annual billing with auto-renewal. Support for coupons and promotional pricing.
  • Free Trials: 14-day trial with no payment upfront. Convert to paid or cancel at trial end.
  • Family Plans: Primary subscriber can invite family members who inherit tier benefits.
  • Invoice and Receipts: Generate PDF invoices, email receipts, and expose billing history via API.
  • Pause Subscription: Allow up to 3-month pause without losing subscriber status.

Non-Functional Requirements

  • Payment Reliability: 99.99% success rate on billing operations. Zero double-charges via idempotency.
  • State Consistency: Subscription state must never diverge between our DB and Stripe. Reconciliation jobs run hourly.
  • Grace Periods: 7-day grace on failed payments before entitlement revocation. Dunning emails at day 1, 3, 5, 7.
  • GDPR Compliance: Right to deletion, data portability for billing history, consent management for marketing.
  • Low Latency Entitlements: Entitlement checks must resolve in <10ms (p99) for every page load and API call.
  • Auditability: Every state transition, payment event, and entitlement change must be immutably logged.
  • Multi-Currency: Support 40+ currencies with daily exchange rate updates. Display local pricing.
  • Scalability: Handle 10x billing spikes on month-start renewal dates without degradation.
PE Signal: Requirement Prioritization In an interview, explicitly state which requirements you would build in V1 vs V2. For V1: subscribe, cancel, billing, entitlement checks. For V2: family plans, pause, coupons, annual billing. This shows you can scope incrementally while keeping the architecture extensible.

Key Capacity Insights

  • Billing is low-throughput, high-correctness: At ~3 TPS average, the challenge is not scale but reliability. Every missed charge is lost revenue; every double charge is a chargeback.
  • Entitlement checks are high-throughput: At 50K QPS, we must cache aggressively in Redis. Cannot hit the DB for every page load.
  • Renewal day spikes: If many subscribers sign up on the 1st of the month, renewal day can see 10x burst. Solution: stagger billing anchors or use a billing queue with rate limiting.
  • Storage is trivial: Subscription data is small. The real storage concern is the immutable billing event log for audit purposes.

Service Responsibilities

Subscription Service

Owns the subscription lifecycle: create, upgrade, downgrade, cancel, pause, resume. Orchestrates calls to Stripe for customer and subscription creation. Publishes domain events to Kafka on every state transition.

Tech: Java/Kotlin, Spring Boot, PostgreSQL

Billing Service

Consumes Kafka events from Stripe webhooks. Manages invoice records, payment status, retry logic, and dunning workflows. Generates PDF invoices and stores in S3. Handles refunds and prorations.

Tech: Java/Kotlin, Spring Boot, PostgreSQL, S3

Entitlement Service

Maps active subscriptions to feature flags and access rights. Maintains a Redis cache for ultra-low-latency lookups. Listens to subscription state change events and updates entitlements. Exposes a gRPC API for internal services.

Tech: Go, Redis, PostgreSQL, gRPC

Notification Service

Sends transactional emails (welcome, receipt, dunning, cancellation) and push notifications. Consumes Kafka events. Manages templates and user communication preferences. Rate limits to prevent spam.

Tech: Python, Kafka, SES, FCM/APNs

PE Signal: Service Boundaries A strong answer separates Subscription (lifecycle), Billing (money), and Entitlement (access) into distinct services. This allows independent scaling, separate on-call rotations, and clear ownership. The Subscription Service is the orchestrator; Billing and Entitlement are reactive consumers of its events.

Critical: Idempotency-Key Header All mutation endpoints (POST, PATCH, DELETE) require an Idempotency-Key header. This prevents double-charges on network retries. The server stores the idempotency key and its response in Redis with a 24h TTL. If the same key is received again, the cached response is returned without re-executing the operation.

From To Trigger Side Effects

TRIALING CANCELLED User cancels during trial Revoke entitlements immediately, no charge, send cancellation email
TRIALING PAST_DUE Trial ends + first payment fails Keep entitlements for grace period, start dunning sequence
ACTIVE ACTIVE Renewal payment succeeds Extend period_end, emit billing event, send receipt
ACTIVE PAST_DUE Renewal payment fails Begin grace period (7 days), start dunning emails, keep entitlements
ACTIVE CANCELLED User initiates cancel Set cancel_at_period_end=true, keep entitlements until period end
ACTIVE PAUSED User requests pause Stop billing, keep limited entitlements, schedule resume_at
PAST_DUE ACTIVE Retry payment succeeds Restore full entitlements, send recovery email, reset retry count
PAST_DUE CANCELLED All retries exhausted (day 7) Revoke entitlements, send final dunning email, offer resubscribe link
PAUSED ACTIVE User resumes or resume_at date arrives Resume billing, restore full entitlements
CANCELLED EXPIRED Billing period ends (for cancel_at_period_end) Revoke all entitlements, final cleanup
CANCELLED ACTIVE User resubscribes within 30 days Create new subscription, carry over loyalty benefits

Grace Period and Dunning Timeline

7-Day Dunning Sequence

Day 0: Payment fails
  -> Status: PAST_DUE
  -> Entitlements: STILL ACTIVE (grace period)
  -> Email: "Your payment failed, please update your card"
  -> Push notification sent

Day 1: First retry (Stripe automatic)
  -> If success: back to ACTIVE
  -> If fail: continue dunning

Day 3: Second retry
  -> Email: "Second attempt failed -- update payment method"
  -> In-app banner: "Action required: payment issue"

Day 5: Third retry
  -> Email: "Your subscription is at risk"
  -> Push notification with deep link to billing page

Day 7: Final retry
  -> If fail: Status -> CANCELLED
  -> Revoke all entitlements
  -> Email: "Your subscription has been cancelled"
  -> Offer: "Resubscribe within 30 days to keep your history"
PE Signal: Why Grace Periods Matter Immediately revoking entitlements on first payment failure leads to terrible UX and increased churn. Most failed payments are recoverable (expired card, insufficient funds on payday). A 7-day grace period with progressive dunning recovers 60-80% of failed payments according to industry data. This is a business-critical design decision.

8 Billing Deep Dive

Stripe Integration Architecture

Stripe Object Mapping

  • Customer: Created 1:1 with Airbnb user. Stores default payment method.
  • Subscription: Mirrors our subscription record. Stripe manages billing schedule.
  • Price: Maps to our plans. One Price per plan+interval+currency.
  • Invoice: Generated by Stripe on each billing cycle. We store a reference.
  • PaymentIntent: Each invoice payment attempt. Contains decline codes on failure.

Our Role vs Stripe's Role

  • Stripe handles: Charge scheduling, retry logic (Smart Retries), card network communication, PCI compliance, receipt emails (optional).
  • We handle: Subscription lifecycle logic, entitlement management, dunning UX, custom retry scheduling, plan catalog, analytics, user-facing billing portal.
  • Overlap: Both track subscription status. We reconcile hourly via Stripe API. Our DB is source of truth for entitlements; Stripe is source of truth for payment state.

Webhook Reliability Pattern

// Stripe webhooks are at-least-once delivery. We must handle:
// 1. Duplicate events (idempotency via stripe_event_id)
// 2. Out-of-order events (check timestamps, use event sequence)
// 3. Missed events (reconciliation job)

class WebhookProcessor {
  async process(event: StripeEvent): Promise<void> {
    // Step 1: Verify signature
    const verified = stripe.webhooks.constructEvent(
      rawBody, signature, WEBHOOK_SECRET
    );

    // Step 2: Idempotency check
    const existing = await db.query(
      `SELECT id FROM billing_events WHERE stripe_event_id = $1`,
      [verified.id]
    );
    if (existing.rows.length > 0) {
      logger.info(`Duplicate event ${verified.id}, skipping`);
      return; // Return 200 to prevent Stripe retry
    }

    // Step 3: Process within a transaction
    await db.transaction(async (tx) => {
      // Insert billing event (idempotency guard)
      await tx.insert('billing_events', {
        stripe_event_id: verified.id,
        event_type: verified.type,
        ...extractEventData(verified)
      });

      // Update subscription state
      await this.handleEventType(tx, verified);
    });

    // Step 4: Publish to Kafka (outside DB transaction)
    await kafka.publish('billing.events', {
      type: verified.type,
      subscription_id: extractSubId(verified),
      timestamp: verified.created
    });
  }
}

Multi-Currency and Tax Handling

Currency Strategy

  • Display Currency: Users see prices in their local currency. Rates updated daily from ECB/Open Exchange Rates API.
  • Billing Currency: Stripe charges in the user's local currency. We create one Stripe Price object per plan per currency.
  • Settlement Currency: Stripe settles in USD to our account. Exchange rate risk is on us for the delta between display and settlement.
  • Price Localization: We round to psychologically friendly prices (e.g., EUR 27.99 instead of EUR 27.43). Prices are manually reviewed quarterly.

Tax Handling

  • Stripe Tax: We use Stripe Tax for automatic tax calculation. It handles VAT (EU), GST (AU/NZ), sales tax (US), and consumption tax (JP).
  • Tax-Inclusive vs Exclusive: EU prices are tax-inclusive (VAT included). US prices are tax-exclusive (tax added at checkout).
  • Tax Receipts: Stripe generates compliant tax invoices. We store the Stripe invoice ID and provide a download link.
  • B2B Reverse Charge: If the subscriber provides a VAT ID, reverse charge applies (0% VAT on cross-border EU B2B).
PE Signal: Reconciliation is Non-Negotiable Webhooks can be lost, delayed, or arrive out of order. A reconciliation job must run hourly, pulling subscription state from Stripe's List Subscriptions API and comparing with our DB. Any discrepancy triggers an alert and auto-correction. For financial systems, "trust but verify" is the operating principle.

// Entitlement check pseudocode:
async function checkEntitlement(userId, entitlementKey) {
  // 1. Check Redis first
  const expiry = await redis.hget(`entitlement:${userId}`, entitlementKey);
  if (expiry) {
    return {
      granted: new Date(expiry) > Date.now(),
      cached: true,
      expires_at: expiry
    };
  }

  // 2. Cache miss: query PostgreSQL
  const ent = await db.query(
    `SELECT * FROM entitlements
     WHERE user_id = $1 AND entitlement_key = $2 AND is_active = true`,
    [userId, entitlementKey]
  );

  if (ent.rows.length > 0) {
    // 3. Populate cache for all user entitlements (batch)
    const allEnts = await db.query(
      `SELECT entitlement_key, expires_at FROM entitlements
       WHERE user_id = $1 AND is_active = true`,
      [userId]
    );
    const hashData = {};
    allEnts.rows.forEach(e => hashData[e.entitlement_key] = e.expires_at);
    await redis.hmset(`entitlement:${userId}`, hashData);
    await redis.expire(`entitlement:${userId}`, 300);
  }

  return {
    granted: ent.rows.length > 0,
    cached: false,
    expires_at: ent.rows[0]?.expires_at
  };
}

Eventual Consistency on Plan Changes

Propagation Timeline

When a subscription state changes (e.g., upgrade from Explorer to Adventurer), entitlements must be updated. Here is the timeline:

  1. T+0ms: User upgrades via API. Subscription Service updates PostgreSQL, publishes subscription.upgraded to Kafka.
  2. T+50ms: Entitlement Service consumes event, updates entitlements table in PostgreSQL (add new, revoke old).
  3. T+60ms: Entitlement Service invalidates Redis cache for this user (DEL entitlement:{user_id}).
  4. T+100ms: Next entitlement check triggers cache miss, repopulates with new entitlements.
  5. Worst case: If cache invalidation fails, old entitlements persist for up to 5 minutes (TTL). This is acceptable because upgrades grant more access (user is never denied something they paid for).
Critical Design Rule: Fail Open for Upgrades, Fail Closed for Downgrades On upgrade, if there is any propagation delay, the user temporarily has their old (lesser) entitlements -- this is safe. On downgrade or cancellation, we must proactively invalidate the cache immediately. If Redis is down, the entitlement check falls back to the database. Never fail open on revocation -- that would give free access.

Cache Invalidation Strategy

  • Event-driven: Kafka consumer in Entitlement Service issues DEL on subscription state changes.
  • TTL-based: 5-minute TTL ensures stale data self-heals even if event is lost.
  • Eager refresh: On user login, proactively refresh entitlement cache.
  • Pub/Sub backup: Redis Pub/Sub channel broadcasts invalidation to all Entitlement Service instances.

Handling Redis Failures

  • Circuit breaker: If Redis is unreachable, trip circuit breaker after 3 consecutive failures.
  • Fallback to DB: Entitlement checks go directly to PostgreSQL read replica.
  • Local cache: 30-second in-memory LRU cache as a final fallback (accepts higher staleness during outage).
  • Monitoring: Alert on cache hit rate drop below 95%. Indicates potential Redis issues.

Entitlement: Sync vs Async Synchronous DB check on every request Async event-driven with Redis cache Async + Redis 50K QPS cannot hit PostgreSQL directly. Redis gives <2ms lookups. Accept 5-min eventual consistency. Use event-driven invalidation for faster convergence.
Consistency Model Strong consistency (two-phase commit with Stripe) Eventual consistency with reconciliation Eventual + Reconciliation 2PC across Stripe boundary is impractical. Use idempotent event processing + hourly reconciliation. Stripe is source of truth for payments; we are source of truth for entitlements.
Entitlement Checks: Centralized vs Distributed Central Entitlement Service (all checks go through one service) Distributed (embed entitlement SDK in each service) Centralized + gRPC Centralized gives single source of truth, easier auditing, and simpler cache invalidation. gRPC keeps latency under 5ms. SDK approach creates cache coherency nightmares at scale.
State Machine: Application vs DB Enforce state transitions in application code Use DB constraints and triggers Application Code Application-level state machine is testable, loggable, and can emit events. DB triggers are opaque and hard to debug. Use partial unique indexes for safety nets.
Billing Anchor: Fixed vs Staggered Bill everyone on 1st of month Bill on subscription anniversary date Anniversary Date Fixed billing creates massive renewal spikes. Anniversary billing distributes load evenly. Stripe supports this natively with billing_cycle_anchor.

Deep Dive: Build vs Buy

Build Custom Billing

  • Full control over billing logic and UX
  • No per-transaction fees to third party
  • Custom retry and dunning strategies
  • Con: PCI DSS Level 1 compliance ($500K+/yr)
  • Con: Card network integrations (Visa, MC, Amex)
  • Con: 12-18 month build time for a dedicated team
  • Con: Tax calculation across 190+ jurisdictions

Use Stripe Billing

  • PCI compliance handled by Stripe
  • Smart Retries (ML-optimized retry timing)
  • Automatic card updater (new card details from networks)
  • Stripe Tax for global tax compliance
  • 2.9% + 30c per transaction (cost)
  • Faster time to market (weeks, not months)
  • Con: Vendor lock-in risk
  • Con: Limited customization of billing flows
PE Signal: Vendor Lock-in Mitigation To mitigate Stripe lock-in, design a Payment Provider Adapter interface. The Subscription Service calls PaymentProvider.createSubscription(), not Stripe.createSubscription(). This allows swapping to Adyen, Braintree, or a custom solution without changing business logic. In practice, migrations are rare, but the abstraction costs almost nothing upfront and buys insurance.

11 Interview Tips and Calibration

Key Signals Interviewers Look For

Strong Hire Signals

  • Separates billing, subscription, and entitlement into distinct services with clear ownership
  • Discusses idempotency for all payment mutations
  • Proposes webhook reliability pattern with deduplication and reconciliation
  • Designs a state machine with explicit transitions and side effects
  • Considers grace periods and dunning as first-class concerns
  • Caches entitlements in Redis with event-driven invalidation
  • Discusses proration, multi-currency, and tax without being prompted

Common Mistakes

  • Treating subscription as a CRUD problem without a state machine
  • No discussion of payment failure handling or retries
  • Checking entitlements directly from the database at 50K QPS
  • Missing idempotency on webhook handler (double-processing events)
  • No reconciliation strategy between local DB and Stripe
  • Over-engineering: building custom payment processing instead of using Stripe
  • Ignoring eventual consistency and its implications for entitlement checks

Follow-Up Questions to Expect

  • "How do you handle a user who upgrades mid-cycle? Walk me through the proration."
  • "What happens if Stripe is down for 2 hours?"
  • "How would you migrate from Stripe to Adyen?"
  • "A user claims they were double-charged. How do you investigate?"
  • "How do you test the billing system? Can you charge real cards in staging?"
  • "What metrics would you monitor on the subscription dashboard?"
  • "How does the family plan work technically?"
